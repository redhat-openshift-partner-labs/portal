## Overview

This application uses a **dual-database architecture**:

| Environment | Database | Schema File | When Used |
|-------------|----------|-------------|-----------|
| Local Development | SQLite | `prisma/schema.prisma` | `pnpm dev` |
| Production | PostgreSQL | `prisma/schema.postgresql.prisma` | Deployed app |

**Why this approach?**
- SQLite requires no external services, making local development fast and portable
- PostgreSQL provides production-grade features, scalability, and reliability
- Prisma ORM abstracts database differences, allowing identical application code

## Type Mapping

Prisma handles type conversion between PostgreSQL and SQLite:

| PostgreSQL Type | SQLite Storage | Prisma Type | Notes |
|-----------------|----------------|-------------|-------|
| `integer` | `INTEGER` | `Int` | Native support |
| `character varying` | `TEXT` | `String` | SQLite uses TEXT for all strings |
| `boolean` | `INTEGER` (0/1) | `Boolean` | Prisma adapter converts automatically |
| `double precision` | `REAL` | `Float` | Both use 64-bit IEEE 754 |
| `timestamp without time zone` | `TEXT` | `DateTime` | SQLite stores ISO8601 strings |
| `GENERATED BY DEFAULT AS IDENTITY` | `AUTOINCREMENT` | `@default(autoincrement())` | Prisma abstracts |

### DateTime Considerations

- **PostgreSQL**: Native timestamp with microsecond precision
- **SQLite**: Text-based storage, typically millisecond precision

**Impact**: For most operations, this difference is negligible. Be cautious with:
- Sub-millisecond timestamp comparisons
- Sorting records created within the same millisecond

## Feature Differences & Resolutions

### 1. Schema Namespace

**Difference:**
- PostgreSQL uses the `portaladmin` schema namespace
- SQLite has no schema namespace support (single namespace)

**Resolution:** The runtime adapter configuration handles this automatically:

```typescript
// server/utils/db.ts
const pool = new Pool({
  connectionString,
  options: `-c search_path="${schema}"`,  // Sets portaladmin
})
```

### 2. Foreign Key Enforcement

**Difference:**
- PostgreSQL enforces foreign keys by default
- SQLite requires explicit `PRAGMA foreign_keys = ON`

**Resolution:** The `@prisma/adapter-better-sqlite3` driver enables foreign keys automatically. No action required.

### 3. Cascade Delete Behavior

The following cascade behaviors are configured identically in both databases:

| Relationship | Cascade Behavior | Effect |
|--------------|------------------|--------|
| `Lab` → `Note` | `ON DELETE CASCADE` | Deleting a lab deletes all its notes |
| `Lab` → `Audit` | No cascade | Audits preserved (referential integrity) |
| `Lab` → `ExtensionRequest` | No cascade | Extension requests preserved |
| `Company` → `Lab` | No cascade | Labs preserved when company deleted |

**In Prisma:**
```prisma
model Note {
  lab Lab @relation(fields: [labId], references: [id], onDelete: Cascade)
}
```

### 4. Auto-Increment Behavior

**Difference:**
- PostgreSQL: `GENERATED BY DEFAULT AS IDENTITY` (allows explicit ID insertion)
- SQLite: `AUTOINCREMENT` (prevents ID reuse after deletion)

**Resolution:** Prisma's `@default(autoincrement())` handles both. For most use cases, this difference is transparent.

## Safeguards

### Schema Validation Script

Run this before committing schema changes:

```bash
pnpm db:validate
```

This compares `schema.prisma` and `schema.postgresql.prisma`, reporting any differences in model definitions.

**What it checks:**
- Identical model names and fields
- Matching field types and attributes
- Consistent relations and constraints

### Runtime Database Selection

The application automatically selects the correct database adapter:

```typescript
// server/utils/db.ts
if (isProduction || databaseUrl.startsWith('postgres')) {
  // Use PostgreSQL with @prisma/adapter-pg
} else {
  // Use SQLite with @prisma/adapter-better-sqlite3
}
```

### Prisma Client Generation

**Critical:** The Prisma client must match the target database:

```bash
# For local development (SQLite)
pnpm prisma generate

# For production deployment (PostgreSQL)
pnpm db:pg:generate
```

Mismatched clients produce this error:
> "The Driver Adapter is not compatible with the provider specified in the Prisma schema."

## Troubleshooting

### "Foreign key constraint failed"

**Cause:** Trying to delete a parent record with dependent children (no cascade).

**Resolution:** Delete child records first, or add `onDelete: Cascade` if appropriate.

### "Driver Adapter not compatible with provider"

**Cause:** Prisma client was generated for the wrong database.

**Resolution:**
```bash
# Regenerate for your target database
pnpm prisma generate          # SQLite
pnpm db:pg:generate           # PostgreSQL
```

### DateTime values differ between environments

**Cause:** Timezone handling or precision differences.

**Check:**
- Ensure all DateTime operations use UTC
- Avoid sub-millisecond precision comparisons
- Verify `timestamp without time zone` interpretation

### Schema mismatch after migration

**Cause:** Only one schema file was updated.

**Resolution:**
1. Update both `schema.prisma` and `schema.postgresql.prisma`
2. Run `pnpm db:validate` to verify sync
3. Create migrations for both:
   ```bash
   pnpm db:migrate --name your_change       # SQLite
   pnpm db:pg:migrate --name your_change    # PostgreSQL
   ```

## Modifying the Schema

### Step-by-Step Process

1. **Edit both schema files**
   - `prisma/schema.prisma`
   - `prisma/schema.postgresql.prisma`
   - Make identical changes to both

2. **Validate synchronization**
   ```bash
   pnpm db:validate
   ```

3. **Create local migration (SQLite)**
   ```bash
   pnpm db:migrate --name add_new_field
   ```

4. **Create production migration (PostgreSQL)**
   ```bash
   DATABASE_URL="postgresql://..." pnpm db:pg:migrate --name add_new_field
   ```

5. **Regenerate Prisma client**
   ```bash
   pnpm prisma generate
   ```

### What to Keep Identical

- Model names and `@@map()` directives
- Field names, types, and `@map()` directives
- Relations (`@relation`)
- Unique constraints (`@unique`, `@@unique`)
- Default values (`@default`)

### What Can Differ

- `datasource` block (provider and url)
- `generator` block (if needed for provider-specific features)

### Adding Indexes

Indexes can be added to both schemas for performance:

```prisma
model Lab {
  // ... fields

  @@index([state])
  @@index([companyId])
  @@index([startDate])
}
```

SQLite and PostgreSQL both support B-tree indexes through Prisma's `@@index` directive.

## Integration Tests

The project includes automated tests that verify database compatibility between SQLite and PostgreSQL:

```bash
pnpm test
```

### What the Tests Verify

| Test Category | What It Checks |
|---------------|----------------|
| **Boolean Handling** | `true`/`false` values stored and retrieved correctly |
| **DateTime Handling** | Round-trip consistency, epoch dates, query comparisons |
| **Foreign Key Cascade** | `Note` records cascade delete when `Lab` deleted |
| **Foreign Key Constraints** | `Audit` and `ExtensionRequest` prevent orphan parent deletion |
| **Float Precision** | Small, large, and typical decimal values |

### Test Files

- `tests/setup.ts` - Test database setup/teardown utilities
- `tests/integration/database-compatibility.test.ts` - Compatibility tests

### Running Tests

```bash
pnpm test           # Run all tests
pnpm test:watch     # Watch mode for development
pnpm test:coverage  # Generate coverage report
```

## Summary

The dual-database architecture provides:

- **Portability**: No external database needed for local development
- **Production parity**: Identical schema in both environments
- **Safety**: Validation script catches schema drift
- **Flexibility**: Easy switching via environment variable

When in doubt:
1. Always edit both schema files
2. Run `pnpm db:validate` before committing
3. Test locally with SQLite before deploying to PostgreSQL
4. Check `server/utils/db.ts` for runtime behavior questions

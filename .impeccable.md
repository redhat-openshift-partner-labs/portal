# Design Context

## Users

**Who**: Red Hat partners (technical and semi-technical users) managing OpenShift cluster lab reservations.

**Context**: Users access this dashboard to request, monitor, and manage lab environments for demos, testing, or training. They need to quickly understand resource status, request extensions, and track costs.

**Job to be done**: Efficiently manage lab reservations without friction — request resources, see status at a glance, handle approvals/denials, and extend reservations when needed.

## Brand Personality

**Voice**: Professional, clear, and direct. No jargon unless domain-appropriate (OpenShift/Kubernetes terms are expected).

**Tone**: Confident but not cold. Helpful without being patronizing.

**3-Word Personality**: Professional, Reliable, Technical

**Emotional Goals**: Users should feel **confidence and trust** — secure that lab resources are managed properly, that the system is reliable, and that their requests are handled transparently.

## Aesthetic Direction

**Visual Tone**: Clean, functional, enterprise-grade. Aligned with Red Hat's design language.

**Primary Reference**: Red Hat Console / PatternFly — stay consistent with Red Hat's existing design system and user expectations.

**Theme**: Both light and dark mode fully supported.

**Colors**:
- Primary: Red Hat Red (#EE0000)
- Semantic colors aligned with PatternFly (success, warning, danger, info)
- Muted grays for backgrounds and secondary text

**Typography**:
- Red Hat Display (headings)
- Red Hat Text (body)
- Red Hat Mono (code/technical)

**Anti-References** (what to avoid):
- Generic SaaS templates — no cookie-cutter Bootstrap/Tailwind UI kit looks
- Consumer/playful apps — no gamification, bright gradients, or casual vibes
- Designs that don't feel like they belong in the Red Hat ecosystem

## Design Principles

1. **Clarity over cleverness**: Information hierarchy should be immediately obvious. Users should never wonder "what does this mean?" or "where do I click?"

2. **Density with purpose**: Enterprise users expect information-rich interfaces, but every element must earn its space. No visual noise.

3. **Consistent with Red Hat**: Follow PatternFly patterns where applicable. Users familiar with Red Hat Console should feel at home.

4. **Accessible by default**: WCAG 2.1 AA compliance. Never rely on color alone — use icons and patterns. Respect reduced motion preferences. Support high contrast modes.

5. **Fast and responsive**: The interface should feel snappy. Loading states should be clear but unobtrusive. Actions should provide immediate feedback.

## Accessibility Requirements

- **Standard**: WCAG 2.1 AA compliance
- **Color**: Never use color as the only means of conveying information; always pair with icons, text, or patterns
- **Motion**: Respect `prefers-reduced-motion`; minimize animations or provide alternatives
- **Contrast**: Support system high-contrast preferences
- **Navigation**: Full keyboard navigation support; visible focus indicators
- **Screen readers**: Semantic HTML, proper ARIA labels, meaningful alt text
